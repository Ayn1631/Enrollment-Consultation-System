# 技术路线

说明：仅对已在当前代码中落地的模块进行打勾；未打勾表示未完成或仅部分实现。

## 0. RAG 与 Agent 技术栈（已定）

- [x] 已落地 `LangChain` RAG 编排、本地 `BM25Retriever` 与依赖缺失时的混合检索降级。
- [x] 已落地 `LangGraph` 工作流编排，以及不可用时的本地顺序回退。
- [x] 已接入 `Neo4j` 图谱增强，并在未配置或异常时自动跳过、不阻塞主流程。
- [x] 已接入 `LangChain4j` HTTP Bridge，并在 Bridge 不可用时回退本地技能执行。

- RAG 编排采用 `LangChain`：
  - 本地模式使用 `LangChain BM25Retriever` 作为首选检索器。
  - 异常或依赖缺失时自动降级为项目内置混合检索（BM25 + 向量 + 关键词 + RRF）。
- Agent 编排采用 `LangGraph`：
  - 使用 `StateGraph` 规划功能执行顺序（`rag -> web_search -> skill_exec -> use_saved_skill -> citation_guard`）。
  - LangGraph 不可用时回退到同优先级本地排序策略。
- 知识图谱增强采用 `Neo4j`：
  - 通过全文索引检索关联节点摘要，将图谱事实作为额外证据块并入 RAG 上下文。
  - Neo4j 不可用时只影响图谱增强，不影响主流程回答。
- 跨语言技能执行采用 `LangChain4j`：
  - 通过 HTTP Bridge 调用 Java 侧 LangChain4j 服务执行历史技能。
  - Bridge 不可用时自动回退本地技能执行，不阻塞会话。

## 1. 数据处理阶段

### 1.1 大小端模式（Small2Big）
- [x] 已实现 `small chunk` / `big chunk` 双层切分，以及通过 `parent_id` / `parent_text` 回溯大块上下文。
- 将文档切分为两层：`small chunk` 与 `big chunk`。
- 检索时先命中 `small chunk`，再通过 `parent_id` 回溯对应 `big chunk`，兼顾精准定位与上下文完整性。
- 推荐切分参数：
  - `small chunk`: 300-500 字，overlap 50-80 字。
  - `big chunk`: 1200-2000 字，按章节或条款组织。

### 1.2 查询召回（Query Expansion for Recall）
- [x] 已实现为 `small chunk` 生成 2-3 条辅助查询、基础质检去重，并接入稀疏/精确召回索引视图。
- 为每个 `small chunk` 生成 2-3 条指向性查询语句，用于增强稀疏检索召回能力。
- 生成查询只作为辅助索引，不替代原文索引，避免语义漂移导致误召回。
- 查询语句需要离线质检，过滤低质量扩展查询。

### 1.3 数据治理
- [x] 已实现统一元数据 Schema 抽取、段落级去重与来源追踪。
- 建立统一数据 Schema：`doc_id/source_url/source_title/publish_date/grab_date/topic/chunk_id/chunk_text`。
- 做段落级去重：`source_url + normalized_text_hash`，避免重复内容污染召回排序。
- 保留来源元数据，保证回答可引用、可追溯。

## 2. 提问阶段

### 2.1 查询改写
- [x] 已实现基础查询改写、硬约束保留与“是否需要改写”判断。
1. 去除口语和无关噪声，保留核心意图。
2. 结合历史上下文和记忆改写查询，但必须保留关键约束（时间、省份、专业、分数、金额、否定词）。
3. 先做改写必要性判断，不必要则不改写，防止越改越偏。

## 3. 召回阶段

### 3.1 多路召回模式
- [x] 已实现向量召回、BM25 召回、关键词精确匹配三路并行召回，并与 Small2Big 联动。
- 同时使用三路召回：
  - 向量召回（semantic）
  - BM25 召回（lexical）
  - 关键词精确匹配（exact match）
- 与 Small2Big + 查询召回联合使用，提升复杂问句命中率。

### 3.2 向量相似度规范（已定）
- [x] 已实现向量 `L2 normalization` 与 `cosine similarity` 主排序。
- 向量统一执行 `L2 normalization`。
- 相似度统一使用 `cosine similarity` 作为线上主排序指标。
- 不使用 `L2 distance` 作为线上主指标。

### 3.3 融合策略
- [x] 已实现 RRF 融合，并在召回后截取 TopN 进入重排阶段。
- 多路召回结果使用 RRF（Reciprocal Rank Fusion）融合，不直接混合原始分数。
- 先取 TopN（如 30-50）进入重排阶段。

## 4. 重排阶段

### 4.1 重排模型
- [x] 已实现重排器，并在模型不可用时自动回退启发式重排。
- 使用重排模型对候选结果重排序，输出 TopK 高置信证据段。
- 第一阶段先用通用重排模型建立基线；第二阶段根据评测收益决定是否微调。

### 4.2 微调策略
- 当硬样本集规模和收益满足门槛（如准确率提升 > 3%）时再做重排微调。
- 微调目标是提升“证据相关性”，而不是优化语言风格。

## 5. 生成阶段

### 5.1 生成策略
- [x] 已实现“基于证据生成”和“证据不足时保守答复”的基础链路。
- 生成必须以检索证据为依据，关键结论强制附来源引用。
- 对证据不足或冲突问题，明确给出“不确定 + 建议咨询官方渠道”答复。

### 5.2 LoRA 微调策略
- LoRA 主要用于输出结构、引用格式、拒答规范，不用于“背知识”。
- 事实正确性优先由 RAG 检索保证，而不是靠参数记忆。

### 5.3 边界约束
- [x] 已落地系统边界提示词与引用校验兜底，完整输入/输出安全审核尚未完成。
- 先采用“规则 + 审核 + 引用校验”三层防线：
  - 系统边界提示词
  - 输入/输出安全审核
  - 引用一致性检查
- 强化学习作为后续增强项，不作为 MVP 阶段前置条件。

## 6. 记忆机制

### 6.1 短期记忆
- [x] 已实现短期记忆读写，并在网关对话链路中接入。
- 使用近期对话窗口作为短期记忆，保障多轮连贯性。

### 6.2 长期记忆
- [x] 已实现滚动长期摘要（`rolling_summary`）的增量更新与上下文接入。
- 对全局历史构建滚动摘要，增量更新，不做全量重复摘要。
- 为每条长期记忆维护 `confidence/source/last_verified/expires_at`。

### 6.3 特殊记忆
- [x] 已实现用户回答风格偏好的 special memory 提取、存储与上下文接入。
- 存储用户偏好、长期目标、沟通风格等个性化信息。
- 支持动态更新、显式删除、过期清理，遵循最小化存储原则。

## 7. 技能保存（全自动治理）

- [x] 已实现技能内容哈希版本化、自动激活最优版本，以及保存/执行/列举基础能力。

- 技能由系统与模型自动生成、自动管理，不依赖人工手动版本控制。
- 采用自动版本治理机制：
  - 内容哈希变化自动生成新版本。
  - 每次发布自动快照（Prompt、工具权限、配置、评测结果）。
  - 自动回归评测，不达标不激活。
  - 自动灰度发布（canary/shadow），异常自动回滚。
  - 长期未使用技能自动降级或归档（TTL）。
- 所有技能全链路审计可追踪，保证可复现、可回退。

## 8. 特殊技能与 MCP

- 引入 MCP 与 Skill 体系，支持联网搜索、网页阅读、外部工具调用。
- 工具调用必须受控：白名单、参数校验、超时、重试、熔断、审计日志。
- 优先本地知识库回答，只有在“强时效问题”时才触发联网增强。

## 9. 评测与实验设计

### 9.1 核心评测指标
- 检索：`Recall@K`、`MRR@K`、`nDCG@K`
- 生成：答案准确率、引用命中率、幻觉率
- 工程：首字延迟、P95 总时延、错误率

### 9.2 消融实验（Ablation）
- 查询改写：开/关
- 查询召回：开/关
- Small2Big：开/关
- 重排模型：开/关
- 融合策略：RRF vs 单路召回

### 9.3 上线门槛
- [x] 已实现离线评测脚本与发布门禁脚本。
- 未通过离线评测与线上灰度指标，不得全量上线。
- 关键指标下降触发自动回滚与告警。
